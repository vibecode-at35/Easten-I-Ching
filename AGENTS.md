# AGENTS.md — Operating Rules for AI Coding Agents

This file governs how any AI agent works in this repository. Read it fully before writing code. These rules override convenience, speed, and your own assumptions.

---

## 1. Golden rules (non-negotiable)

1. **The model never computes the hexagram.** All casting, hexagram identification, changing-line detection, and resulting-hexagram computation is **deterministic TypeScript**, validated against `docs/ICHING_REFERENCE.md`. LLMs are unreliable at this symbolic computation and will hallucinate. No exceptions.
2. **Never invent I Ching content.** Hexagram names, judgments, images, line texts, and the King Wen mapping come *only* from `docs/ICHING_REFERENCE.md` and the seeded corpus in the database. If a value is missing, stop and flag it — do not generate it from memory.
3. **Interpret the question, not just the hexagram.** A reading must be specific to the user's actual situation. Dumping generic hexagram text is a failure. The LLM synthesizes primary hexagram + changing lines + movement toward the resulting hexagram into guidance for *this* question.
4. **Ground every LLM call in verified texts.** The interpretation service fetches exact texts from the database and passes them to the model. The model receives source text; it does not recall it.
5. **No dark patterns. Ever.** No manipulative upsells, no fear-based copy, no ads, no fake scarcity. Seriousness and respect for the user are the brand. This is a product requirement, not a preference.
6. **Build only the active task.** Implement the current milestone in `docs/TASKS/` — nothing more. Do not scaffold future features, add speculative abstractions, or "while I'm here" extra work. If something seems necessary but is out of scope, note it and ask.

---

## 2. Scope discipline

- One task at a time, from `docs/TASKS/`. The active task defines what "in scope" means right now.
- If a task is ambiguous or you discover it depends on undecided design, **stop and ask** rather than guessing.
- Do not expand surface area: no new dependencies, services, or config without a clear reason tied to the active task. Prefer the standard stack in `docs/ARCHITECTURE.md`.
- When you finish a task, **stop and report** what you built, what you tested, and what's next — do not roll straight into the next milestone.

---

## 3. Coding conventions

- **Language:** TypeScript everywhere. Strict mode on. No `any` without a written reason.
- **Frontend:** Next.js (App Router), React function components, Tailwind. Keep components small and readable.
- **Backend:** Next.js API routes or a thin Node/TS service. Keep it monolithic for the MVP — no premature microservices.
- **Pure logic is pure.** The casting engine and any domain math live in framework-free modules with no I/O, so they're trivially testable.
- **Determinism is testable.** Anything described as "deterministic" must have unit tests. The casting engine ships with tests that validate against the King Wen table in `docs/ICHING_REFERENCE.md`.
- **Model access is behind an interface.** All LLM calls go through a single `interpretation`/model-router module so models and tiers can be swapped without touching feature code. Never scatter raw API calls through the codebase.
- **Secrets:** environment variables only. Never commit keys. Never log full prompts containing user data in production.
- **i18n from the start:** user-facing strings are localizable (vi / zh / en). Don't hardcode display text.

---

## 4. Domain handling (I Ching)

- The canonical encoding, trigram definitions, casting probabilities, and King Wen mapping are in `docs/ICHING_REFERENCE.md`. Treat that file as the source of truth.
- The King Wen table in that file is a **draft pending verification**. Any milestone that implements the casting engine must include a test that validates hexagram identification against an authoritative published source (see the reference doc). Do not ship the engine on the unverified table alone.
- The corpus (judgments, images, line texts, commentary) is data, loaded into Postgres — not hardcoded in logic, not generated by a model.

---

## 5. Product & safety guardrails

- Readings are **reflective, not prescriptive prophecy.** The voice guides thinking; it does not predict fixed outcomes or claim certainty. See `docs/INTERPRETATION_PROMPT.md`.
- For weighty topics (health, legal, financial, self-harm, crisis), the product encourages reflection and points toward qualified human help; it never poses as a substitute. Bake this into the system prompt, not as an afterthought.
- Respect privacy: a user's questions and journal are sensitive. Minimize logging, scope data access, and never expose one user's readings to another.

---

## 6. Definition of done (every task)

A task is done only when:

- [ ] It does exactly what the task file specifies — no more, no less.
- [ ] All deterministic logic has passing unit tests.
- [ ] Code is typed, runs, and is readable by a human reviewer.
- [ ] No secrets, no dark patterns, no invented I Ching content.
- [ ] You've reported what was built/tested and stopped for review.

When in doubt, choose correctness and honesty over cleverness. Flag uncertainty rather than papering over it.
